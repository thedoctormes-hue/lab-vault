# ADR-007: Проекты и Project Tokens

## Статус
Accepted

## Дата
2026-06-12

## Автор
ant (по запросу ЗавЛаба)

## Контекст

ЗавЛабу нужен способ передавать секреты лаборантам без компрометации в чате. Существующий механизм SecretToken решает проблему для одного секрета, но не покрывает сценарий «передать набор секретов для проекта».

**Проблема:** лаборанту нужно получить несколько секретов (db_pass, api_key, smtp_pass) для деплоя приложения. Способы передачи:

1. **Каждый секрет отдельным токеном** — лаборант делает N запросов, N токенов в чате
2. **Секреты в открытом виде в чате** — компрометация, ❌
3. **Один токен на все секреты проекта** → выбранный вариант

**Требования:**
- Один токен = все секреты проекта разом
- Токен одноразовый (сгорает после использования)
- Секрет не появляется в чате в открытом виде
- Управление через TG-бот (создание проектов, добавление секретов, генерация токенов)
- Поддержка записи секретов в .env файл для деплоя

## Решение

Ввели две новые сущности: **Project** и **ProjectToken**.

### Project

Группа секретов с общим именем и ID.

```go
type Project struct {
 ID string // уникальный slug-идентификатор
 Name string // человекочитаемое имя
 SecretIDs []string // список имён секретов (N:M)
 CreatedAt time.Time
}
```

- Хранится в `Config.Projects map[string]*Project`
- Персистится в config.yaml под ключом `projects`
- Один секрет может входить в несколько проектов

### ProjectToken

Одноразовый токен доступа ко всем секретам проекта.

```go
type ProjectToken struct {
 ProjectID string // ID проекта
 Token string // SHA-256 хеш
 CreatedAt time.Time
 ExpiresAt time.Time // zero = бессрочно
 Revoked bool
}
```

- Генерируется через `POST /project-tokens/:project_id` или через TG-бот
- Оригинал токена показывается один раз, в config.yaml — SHA-256 хеш
- One-time: отзывается после первого успешного использования
- TTL наследуется от `token_ttl_hours`

### HTTP API

| Метод | Путь | Описание |
|-------|------|----------|
| GET | `/projects` | Список проектов |
| POST | `/projects` | Создать проект |
| GET | `/project/:id` | Получить проект |
| DELETE | `/project/:id` | Удалить проект и его токены |
| GET | `/project-tokens/:project_id` | Токены проекта |
| POST | `/project-tokens/:project_id` | Создать токен |

### `/access/:token` — расширение

Эндпоинт теперь поддерживает оба типа токенов:

1. **SecretToken** → `{name, value, updated_at}` (как было)
2. **ProjectToken** → `{project, project_id, secrets: {name: {name, value, updated_at}}}`

Определение типа токена: сначала проверяем `SecretTokens`, затем `ProjectTokens`.

### lab-vault-env — расширение

Добавлен флаг `--write-to <file>` для записи секретов проекта в .env файл:

```bash
lab-vault-env -token <project-token> --write-to /path/to/.env
```

Формат .env: `NAME=value\n`, значения экранируются (newlines → `\n`, спецсимволы в кавычках).

### TG-бот — расширение

**FSM-состояния:**
- `stateWaitingProjectID` → `stateWaitingProjectName` → `stateWaitingProjectSecrets`
- `stateWaitingAddSecretName` — добавление секрета в существующий проект
- `stateWaitingReplaceSecrets` — замена набора секретов проекта

**Кнопки в карточке проекта:**
- 🔑 Создать токен
- ➕ Добавить секрет
- ✏️ Заменить секреты
- 🗑 Удалить проект

## Последствия

**Положительные:**
- Один токен = N секретов, удобно для деплоя
- Секреты не появляются в чате
- Токен одноразовый — перехват бесполезен после использования
- .env файл генерируется одной командой
- N:M между секретами и проектами — гибкость

**Отрицательные:**
- Project token даёт доступ ко ВСЕМ секретам проекта — нет гранулярности
- O(n) поиск по project tokens (нет индекса по ProjectID)
- Усложнение FSM бота (5 новых состояний)

**Риски:**
- Утечка project token = утечка всех секретов проекта (blast radius)
- Митигация: one-time токены + TTL

## Альтернативы (отклонены)

1. **Роли и ACL** — избыточно для текущего масштаба, добавим позже если нужно
2. **Шифрование в чате** — сложнее в реализации, UX хуже
3. **Один токен на все секреты без проектов** — нет группировки, нельзя управлять наборами
